Summary:
Implemented orchestration resilience guards for duplicate/retry-safe command dispatch with actionable reason codes.

Files touched:
- orchestration-server/app/main.py
- orchestration-server/app/models.py
- orchestration-server/app/registry.py
- orchestration-server/README.md
- docs/TICKETS.md

Behavior change:
- Added request-level idempotency for operator command endpoints and `play_at` via optional `request_id`.
- Repeated requests with the same `request_id` replay cached response instead of redispatching (`idempotent_replay=true`, `reason_code=DUPLICATE_REQUEST`).
- Node-level enqueue now rejects stale/duplicate `seq` values (`reason_code=DUPLICATE_OR_STALE_SEQ`), preventing duplicate node actions.
- Dispatch responses now include per-node `reason_code` details.
- `PLAY_AT` validation error now returns structured actionable payload:
  - `reason_code=PLAY_AT_LEAD_TIME_TOO_SHORT`
  - `min_lead_ms`

Contract impact: additive

Sanity gate result: pass

- .sanity-logs/report.txt

How to test:
1) Run `make sanity`.
2) Register a node: `POST /api/v1/nodes/register`.
3) Call `POST /api/v1/operators/pause` twice with same `request_id` and target node.
   - first call dispatches normally
   - second call returns same `seq` with `idempotent_replay=true` and `reason_code=DUPLICATE_REQUEST`
4) Call `POST /api/v1/shows/play_at` with too-small lead time and verify `422` with reason payload.

Risks/assumptions:
- Idempotency cache is in-memory with 120s TTL; not shared across processes.
- Full distributed idempotency will require persistent cache (Redis/Postgres) in a later phase.
